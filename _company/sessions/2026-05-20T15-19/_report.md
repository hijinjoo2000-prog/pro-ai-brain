# 🛠️ Network Error Diagnosis Report: Failed to fetch
**진단 목표:** `lastError: "network"`, `lastErrorMessage: "Failed to fetch"`의 근본 원인 진단 및 해결책 제시.
**작성자:** 코다리 (시니어 풀스택 엔지니어)

## 📄 개요 분석 (Initial Assessment)
`Failed to fetch`는 HTTP 상태 코드(예: 401, 503)를 받기 전에 네트워크 통신 자체가 끊기거나 차단되었을 때 발생하는 브라우저 API 레벨의 오류입니다. 즉, **문제의 위치가 클라이언트-브라우저, 중간 네트워크 레이어(프록시/방화벽), 또는 서버 인바운드 설정 중 어디든 있을 수 있습니다.**

---

## 💡 진단된 근본 원인 (3+ Root Causes)

### 1. CORS 정책 위반 (Cross-Origin Resource Sharing Violation)
가장 흔한 원인입니다. 클라이언트(프론트엔드)의 도메인이 서버(백엔드 API)에서 기대하는 출처와 다를 경우, 보안상의 이유로 브라우저가 요청 자체를 차단합니다.

*   **진단 근거:** `Failed to fetch`는 CORS Preflight 요청 (`OPTIONS` 메서드) 자체가 실패했거나, 서버가 응답 헤더(`Access-Control-Allow-Origin`)를 적절히 설정하지 못했을 때 발생하기 쉽습니다.
*   **기술적 방안:** 백엔드 API 게이트웨이나 웹 서버(Nginx/Apache) 레벨에서 Origin을 허용하는 설정을 반드시 추가해야 합니다.

### 2. 네트워크 또는 보안 레이어 차단 (Network or Security Layer Blocking)
클라이언트와 서버 사이에 존재하는 장비(방화벽, VPN, 프록시, CDN 등)가 요청의 패킷 자체를 악성으로 간주하고 차단할 때 발생합니다. 이는 API 엔드포인트나 포트 접근에 대한 IP 기반 제한일 수 있습니다.

*   **진단 근거:** 로컬 개발 환경이나 특정 사내망에서만 재현되는 경우, 또는 Public CDN을 사용할 때 갑자기 문제가 생기는 경우를 의심해야 합니다.
*   **기술적 방안:** 요청 경로가 열려 있는지 확인하고, 필요한 헤더(특히 인증 토큰이 포함된 `Authorization` 헤더)가 차단되지 않도록 설정합니다.

### 3. 백엔드 서버의 비가용성 또는 라우팅 오류 (Backend Unavailability / Routing Error)
API를 호출하려는 엔드포인트 자체가 다운되었거나, 요청을 받을 수 없는 상태일 때 발생할 수 있습니다. 특히 Docker 컨테이너나 배포 환경에서 포트 매핑이 잘못되었거나, 서버가 초기화되지 않은 경우에 해당합니다.

*   **진단 근거:** `lsof -nP -iTCP:PORT` 명령어로 확인했을 때 포트는 열려 있지만, 실제로 요청을 처리할 프로세스가 죽어있거나(Crashed), 로드 밸런서 뒤에서 라우팅 규칙이 실패한 경우입니다.
*   **기술적 방안:** 백엔드 로그를 확인하여 서버가 정상적으로 시작했는지, 그리고 요청된 경로(Route)에 대한 핸들러가 존재하는지 검증해야 합니다.

## 🚀 즉각적인 디버깅 단계 (Immediate Action Plan)

1.  **브라우저 개발자 도구 활용:** 크롬/파이어폭스 개발자 도구의 **Network Tab**을 열고, 요청 시도를 재현합니다.

3.  **Console Log 확인:** 콘솔 창에 출력되는 상세 에러 메시지 (예: "No 'Access-Control-Allow-Origin' header is present...")를 통해 정확한 원인을 특정할 수 있습니다.
