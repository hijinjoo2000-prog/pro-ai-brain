# 🏗️ 기술 통합 전제 조건 및 표준 API 연동 가이드라인 리포트
**작성 목적:** 공인중개사 대상 '대한민국 재개발 No.1 플랫폼' 구축을 위한 실시간, 오차 없는 데이터 병합 아키텍처 설계 및 필수 체크리스트 제공.

## Ⅰ. 핵심 과제 분석: 왜 최신 아키텍처가 필요한가?
공동 목표로 설정된 세 가지 데이터 소스는 성격이 완전히 다릅니다.
1. **네이버 매물:** 비정형, 실시간 변동성 높음 (가격, 매물 상태).
2. **법제처/정보몽땅:** 정형적, 주기적 변화, 법규 변경에 민감함 (절차, 구역 지정).
3. **데이터 병합 요구사항:** 이 세 가지 데이터가 동시에 특정 시점(T)의 '법적 위험도'와 '시세 적정성'을 판단하는 데 사용되어야 합니다.

단순한 CRUD 기반의 웹 서비스로는 이러한 복잡하고 상호 의존적인 데이터를 실시간으로 통합하고, 한 정보의 변경이 다른 정보에 미치는 **파급 효과(Impact)**를 추적할 수 없습니다. 따라서 데이터의 변화 자체를 핵심 가치로 삼는 아키텍처가 필수입니다.

## Ⅱ. 추천 백엔드 아키텍처 패턴 (Data Merging Layer)

| 패턴 | 정의 및 역할 | 왜 재개발 플랫폼에 필요한가? |
| :--- | :--- | :--- |
| **1. 이벤트 소싱 (Event Sourcing)** | 애플리케이션의 상태 변화를 '이벤트'라는 불변의 순서 기록(Log)으로 저장하는 방식. 현재 상태 대신, *어떻게* 그 상태에 도달했는지를 추적합니다. | 재개발 과정은 시간의 흐름에 따른 법적 절차의 누적입니다. (예: **[토지이용계획변경_승인됨]** $\rightarrow$ **[조합설립추진위_구성완료]**). 이벤트 소싱을 통해 모든 '법적 상태 변화'를 추적하고, 특정 시점 T-1의 법률을 재현할 수 있습니다. |
| **2. 마이크로서비스 아키텍처 (Microservice Gateway)** | 시스템을 기능별로 독립적인 작은 서비스들(마이크로서비스)로 분리합니다. 각 서비스는 자체 데이터베이스를 가지며, API 게이트웨이가 이를 통합하여 사용자에게 제공합니다. | 세 개의 핵심 데이터 소스(매물, 법규, 구역진행상황)는 독립적입니다. 각각의 데이터를 **전문적으로 처리하는 전용 마이크로서비스**를 분리해야 합니다. (예: `PriceService`, `LegislationService`, `StatusService`). |
| **3. CQRS (Command Query Responsibility Segregation)** | 데이터 쓰기(Write/Command) 모델과 읽기(Read/Query) 모델을 분리합니다. | *쓰기:* 이벤트 소싱 기반으로 원천 데이터를 기록하고 검증하는 곳입니다. *읽기:* 공인중개사에게 보여줄 최종 '진단 보고서' 형태로 최적화된 쿼리를 제공하는 전용 API 레이어를 구축할 때 효율적입니다. (성능 및 보안 강화). |

**✅ 결론:** 세 가지 패턴을 혼합한 **이벤트 소싱 기반의 마이크로 서비스 게이트웨이 구조(Event-Sourced Microservices Gateway)**를 채택해야 합니다.

## Ⅲ. API별 통합 기술 체크리스트

| 구분 (데이터 소스) | 데이터 유형 | 필수 아키텍처 패턴 | 인증/접근 방식 | Rate Limit 고려사항 |
| :--- | :--- | :--- | :--- | :--- |
| **1. 네이버 매물 크롤링** | 비정형, 가격 변동 | *데이터 정제 서비스* (Scraping & Cleaning Service) $\rightarrow$ **이벤트 소싱 적용** | API Key/User-Agent Spoofing (기술적 우회 필요). 법적 고지 필수. | IP 기반 제한(IP Blocking) 및 시간당 요청 횟수 체크. 지연 호출(Exponential Backoff) 로직 구현 필수. |
| **2. 법제처 재개발 절차법** | 정형, 법규/절차 | *레거시 데이터 서비스* (Legislation Service) $\rightarrow$ **CQRS 적용** | 공공 API Key (OAuth 2.0 권장). 접근 토큰 만료 주기 체크. | 정부 기관의 표준 Rate Limit(예: 분당 10회 등) 준수. 배치(Batch) 요청 최적화 필요. |
| **3. 정보몽땅 구역별 진행상황** | 정형, 현황/지표 | *상태 관리 서비스* (Status Service) $\rightarrow$ **이벤트 소싱 적용** | 공공 API Key 또는 전용 데이터 연동 계약 검토. | 특정 지역(구역 ID)에 대한 과부하 방지를 위한 요청 분산(Load Balancing) 설계 필요. |
| **4. 통합 게이트웨이 (API Gateway)** | 최종 사용자 인터페이스 | 마이크로 서비스 게이트웨이 (Microservice Gateway) | 모든 API 호출은 공인중개사 로그인 인증 기반으로 처리해야 함. | 종합적인 요청 검증(Validation), 속도 제한(Throttling), 캐싱 계층 도입 필수. |

## Ⅳ. 기술 구현 액션 플랜 요약 (Next Steps Blueprint)

1. **[Backend] 이벤트 스트림 구축:** Kafka 또는 AWS Kinesis와 같은 메시지 브로커를 중심으로 데이터 변경 이벤트를 발행/구독하는 시스템을 설계합니다.
2. **[Service Layer] 3개 전용 마이크로서비스 개발:** 매물, 법규, 구역 상태에 대한 독립적인 데이터 수집 및 정제 로직을 각각의 서비스(Microservice)로 구현합니다.
3. **[API Gateway] 통합 및 인증 계층 구축:** 모든 외부 요청은 게이트웨이를 통과하며, 여기서 Rate Limit, 사용자 권한 체크, 그리고 최종 데이터를 조합하는 '진단 알고리즘 실행'이 이루어지도록 설계합니다.
