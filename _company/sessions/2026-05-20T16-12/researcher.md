## 🛠️ 재개발 정보 데이터 통합 아키텍처 블루프린트 (V1.0)

### 🎯 분석 목표
네이버 매물정보, 법제처 API, 정보몽땅 API라는 성격이 완전히 다른 세 가지 데이터를 공통의 기준으로 병합(Merge)하기 위한 최소한의 **공통 핵심 데이터 모델(Core Data Model)**을 정의하고, 각 소스에서 해당 데이터를 추출하는 데 필요한 기술적 사양을 정리합니다.

### 💡 핵심 전제 조건 (데이터 통합의 근거)
세 소스의 데이터를 성공적으로 병합하려면, 단순히 '주소'만으로는 부족하며 다음 세 가지 계층 구조를 반드시 확보해야 합니다.

1. **물리적 주소:** 사용자가 검색하는 일반적인 주소 (예: 서울시 강남구 역삼동 123-4)
2. **법적 식별자:** 해당 필지 전체를 고유하게 지칭하는 코드 (필지코드, Lot ID). 이것이 데이터 통합의 *Primary Key*가 됩니다.
3. **법규/행정 상태:** 이 주소와 필지에 적용되는 법률 및 행정구역 정보 (용도지역, 재개발 구역 지정 여부 등).

### 🔑 공통 핵심 필드 그룹 정의 (Core Data Schema)

| 필드명 | 데이터 유형 | 필수 여부 | 설명/포함 내용 | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| `Matter_ID` | String | 필수 | 통합을 위한 고유 식별자 (UUID 또는 결합 키) | 모든 소스에서 생성되어야 함. |
| `Standard_Address` | String | 필수 | 법적 주소 표기 (예: 강남구 역삼동 OOO-O) | 표준화된 텍스트 형태. |
| `Parcel_Code` | String | 필수 | 필지 고유 식별 코드 (필지코드/Lot ID) | **최우선 매칭 키.** |
| `Zoning_Status` | Code | 필수 | 용도지역 및 지구 지정 상태 코드 | 법제처 API 연동 필요. |
| `Development_Stage` | Enum | 필수 | 재개발 진행 단계 (예: 초기 검토, 조합설립추진, 관리처분인가 등) | 정보몽땅/법제처 간 교차 검증 필요. |
| `Market_Price_Range`| Tuple | 선택 | 현재 매물 기준 시세 범위 (최소-최대) | 네이버 크롤링 결과가 주 역할. |
| `Last_Update_Date` | Date | 필수 | 데이터 수집 및 통합이 완료된 시간 스탬프 | 신뢰성 확보에 중요. |

### 📊 소스별 API 파라미터 및 입력값 정리표

아래 표는 공통 핵심 필드 그룹을 채우기 위해 각 외부 소스가 제공하는 특정 기능(API)과 필요한 최소한의 요청 사양입니다.

| 데이터 소스 | 수집 목표 (핵심 필드) | 권장 API/기술 | 필수 파라미터 예시 | 입력값 및 처리 로직 | 비고 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **네이버 매물정보** | `Standard_Address`, `Market_Price_Range` | 크롤링 / 네이버 API (Property) | `query`: 검색어, `startPage`: 페이지 번호, `areaCode`: 지역 코드 | 1. 주소(검색어) 입력 → 2. 매물 리스트 파싱 → 3. 개별 시세 범위 추출 및 표준화 (`Market_Price_Range`). | **데이터 구조의 비정형성**이 가장 큰 장애물입니다. 크롤링 후 반드시 정제/표준화 모듈 필요. |
| **법제처 API** | `Standard_Address`, `Zoning_Status` | 법령 정보 검색 API (Legal) | `address`: 주소 문자열, `lawType`: 법률 종류, `status`: 조회 상태 | 1. 주소를 기반으로 해당 필지의 **최상위 적용 법규**를 검색 → 2. 용도지역 코드 및 최신 변경 사항을 추출하여 `Zoning_Status`에 할당. | 지번/주소 입력을 통해 가장 정확한 **법적 근거(Legal Basis)**를 확인하는 것이 핵심입니다. |
| **정보몽땅 API** | `Development_Stage`, `Parcel_Code` | 구역 정보 조회 API (Local) | `districtId`: 구역 ID, `matterType`: 구역 종류, `dateRange`: 기간 | 1. 필지코드 기반으로 해당 주택이 속한 재개발 **사업구역의 공식 지정 단계**를 조회 → 2. 이 단계 정보를 `Development_Stage`에 할당. | 이 소스는 '진행상황'이라는 가장 중요한 비정형 데이터를 제공하므로, 다른 두 데이터로 부족한 부분을 보완합니다. |

### ✅ 최종 요약 및 게이트웨이 구현 방향

1. **데이터 통합의 핵심:** 세 가지 필드를 병합하는 데 있어 공통 분모는 **`Parcel_Code` (필지코드)**입니다. 따라서, 사용자가 주소를 입력하면 → `주소(Input)`를 받아 ➡️ `법제처 API`와 `정보몽땅 API`가 각각 `Parcel_Code`를 반환하게 만드는 것이 최우선 구조가 되어야 합니다.
2. **기술적 요구사항:** 코다리 에이전트에게 전달할 명세서에 따르면, 게이트웨이는 단순 호출을 넘어, 각 소스에서 받은 데이터를 위 정의된 Core Data Schema에 맞게 변환(Transformation)하는 **Adapter Pattern** 구현이 필수입니다.

## 🎯 목표: 세 가지 이질적 소스 (네이버, 법제처, 정보몽땅)의 공통 핵심 필드 정의 및 API 사양화

### I. 공통 핵심 데이터 모델 (Core Data Schema)
*   **Matter_ID:** UUID 또는 결합 키
*   **Standard_Address:** 표준 주소 텍스트
*   **Parcel_Code:** **[Primary Key]** 필지 고유 식별 코드
*   **Zoning_Status:** 용도지역 및 지구 지정 상태 (법제처 기반)
*   **Development_Stage:** 재개발 진행 단계 (정보몽땅 기반)
*   **Market_Price_Range:** 현재 시세 범위 (네이버 크롤링 기반)
*   **Last_Update_Date:** 데이터 최종 업데이트 일자

### II. 소스별 필수 파라미터 및 역할 분담
| Source | Key Role | Core Field Contribution | Primary Input/Key Flow |
| :--- | :--- | :--- | :--- |
| **법제처 API** | 법적 근거 제공 | `Standard_Address`, `Zoning_Status` | 주소(Input) $\to$ `Parcel_Code` 획득 및 용도지역 매핑 |
| **정보몽땅 API** | 행정 상태 제공 | `Development_Stage`, `Parcel_Code` | `Parcel_Code` (법제처에서 받은 것) $\to$ 진행 단계 조회 |
| **네이버 크롤링** | 경제적 가치 제공 | `Market_Price_Range` | 주소(Input) $\to$ 매물 검색 및 시세 파싱 |

### III. 게이트웨이 설계 지침 (Gateway Mandate)
1.  **Key Flow:** 모든 요청은 반드시 사용자 입력 주소를 받아 **[법제처 API]**를 통해 `Parcel_Code`를 확보하는 것이 시작점이어야 한다.
2.  **Transformation Layer:** 각 소스별로 데이터를 가져온 후, 표준화된 JSON 포맷으로 변환하는 Adapter 모듈을 구현한다. (예: 네이버의 '최저가'와 법제처의 '법적 용도지역'은 같은 필드에 병합될 수 없음)
